# 仕様書作成ガイドライン

## 1. 目的

個別仕様書の書式、品質基準、変更手続きを統一する。

## 2. 基本原則

### 2.1 仕様先行

実装前に、責務、入出力、エラー、テスト観点を文書化する。

### 2.2 入力不変

各プロセスは入力成果物を直接変更せず、新しい成果物を生成する。

### 2.3 正本を1つにする

同じ内容をMarkdown、YAML、TTS用文字列で保持する場合でも、正式な内容を示す正本を1つ決める。

### 2.4 外部依存を分離する

AI API、VOICEVOX、COEIROINK、ffmpegなどに依存する処理と、パス生成、分割、検証などの純粋処理を分ける。

### 2.5 互換性を段階的に扱う

新形式を優先し、存在しない場合だけ旧形式を読む。旧形式への新規書き込みは段階的に停止する。

## 3. 各仕様書の標準構成

1. 目的
2. 対象範囲
3. 対象外
4. 現行実装
5. 推奨仕様
6. 入力
7. 出力
8. 必須項目
9. 任意項目
10. バリデーション
11. 状態・エラー・警告
12. 正常例
13. 異常例
14. テスト観点
15. 移行・互換性
16. 未決定事項
17. 完了条件

## 4. データ形式の使い分け

| 形式 | 用途 |
|---|---|
| YAML | 人間が編集する企画、設定、プロファイル、承認情報 |
| JSON | 大量処理するマニフェスト、検査結果、機械生成ログ |
| Markdown | 人間が読む原稿、レビュー報告、仕様書 |
| TXT | 既存処理との互換入力、単純なTTS入力 |
| WAV | TTS生成直後の可逆中間音声 |
| MP3 | 章単位または配布用音声 |
| M4B | 将来検討する章情報付きオーディオブック。初期必須出力には含めない |

## 5. 命名

- 仕様書ファイル名は英小文字のkebab-caseとする。
- データ内部のキーは新仕様ではsnake_caseとする。
- 現行のcamelCaseは互換入力として許可する。
- IDはASCII英小文字、数字、ハイフンを基本とする。
- 日本語タイトルと内部IDを分離する。

## 6. MUST・SHOULD・MAY

- **MUST**: 実装が必ず満たす。
- **SHOULD**: 原則として満たす。満たさない場合は理由を記録する。
- **MAY**: 任意で対応できる。

## 7. スキーマの変更

- 後方互換な項目追加はminor変更とする。
- 必須項目の追加、意味変更、削除はmajor変更とする。
- 旧バージョンを読む変換処理を、最低1 major世代は維持する。
- 実装が未知のmajor versionを受け取った場合は処理を停止する。

## 8. テスト要件

各仕様には最低限、次のテストを用意する。

- 最小正常例
- 全項目正常例
- 必須項目欠落
- ID重複
- 未知の列挙値
- 境界値
- 旧形式からの変換
- 入力ファイル不変

外部APIの実呼び出しは統合テストに分離し、通常のpytestではmockを使用する。

## 9. 完了条件

仕様書は、次を満たした時点で`review`へ進める。

- 最小例と完全例がある。
- 前工程と後工程の接続が説明されている。
- 正本と派生物が区別されている。
- エラーと警告が区別されている。
- 既存コードへの影響が記載されている。
- テスト可能な完了条件がある。

## 10. ディレクトリ構成とライフサイクル

仕様・提案・タスクは、次の4ディレクトリへ厳密に区別して格納する。

```text
docs/specifications/   承認済み仕様(製品全体・アーキテクチャ・処理契約・保存方針・配布方針)
docs/db/                承認済みDBテーブル仕様(方針・ERD・個別テーブル定義)
docs/screens/           承認済み画面仕様(画面構成・責務・表示項目・遷移)
docs/spec-proposals/    未承認・provisional・blockedな仕様提案(flat配置)
docs/spec-proposals/task/  未実行の仕様策定タスク
docs/tasks/             未実行の開発・作業タスク(承認済み仕様の実装指示)
```

### 10.1 昇格のライフサイクル

1. 検討中の内容は`docs/spec-proposals/`直下へ提案として作成する。
2. 昇格条件(矛盾がない、`evidence_gap`/`human_review_required`が残っていない、
   製品範囲に必要である等)を満たし、人間が承認した場合のみ、対応する
   承認済みディレクトリ(`docs/specifications/`、`docs/db/`、`docs/screens/`)へ
   移す。
3. 昇格後、`docs/spec-proposals/`側の対応する提案ファイルは削除する。
   承認済み仕様と同じ内容を`docs/spec-proposals/`へ重複して残さない。

### 10.2 禁止する階層

`docs/spec-proposals/generated-specifications/`のような、本ガイドラインに
定義されていない新しいサブディレクトリを作らない。`docs/spec-proposals/`直下は
flat配置とし、既存の`task/`・`examples/`以外の階層が必要になった場合は、
先に本ガイドラインへ追加して人間承認を得る。

### 10.3 タスクディレクトリの区別

```text
docs/spec-proposals/task/   仕様策定タスク(未承認仕様を確定させるための検討)
docs/tasks/                 承認済み仕様に基づく実装タスク
```

いずれも、完了したタスクファイルは削除する。タスクの成果物・実行報告・
意思決定表をタスクディレクトリ自体へ残さない。完了履歴はGit履歴を正本とする。
